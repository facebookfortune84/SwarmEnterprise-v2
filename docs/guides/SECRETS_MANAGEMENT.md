## Overview

SwarmEnterprise v2 uses a single `.env` file for local development.  **This file must never be
committed to version control.**  Staging and production environments load secrets from a secrets
manager (HashiCorp Vault, AWS Secrets Manager, or GitHub Actions Secrets) — never from files on
disk.

```
.env.example  ← schema only, committed, no real values
.env          ← real values, NEVER committed (.gitignore)
```

## Generating Secrets

Use the bundled generator to produce cryptographically strong values with one command:

```bash
python scripts/generate_secrets.py
```

**Sample output:**

```
# 64-char hex — signs and verifies JWT access/refresh tokens — rotate every 90 days
JWT_SECRET_KEY=a3f1c7...

# 64-char hex — signs session cookies, CSRF tokens — rotate every 90 days
SECRET_KEY=9d82bb...

# 32-byte URL-safe — Postgres password — rotate every 180 days
POSTGRES_PASSWORD=X7k9mQ...

# 32-byte URL-safe — symmetric encryption key — rotate every 365 days
ENCRYPTION_KEY=pT4nRv...
```

Copy the `KEY=VALUE` lines into your `.env`, then run validation:

```bash
python scripts/validate_env.py
```

> **Tip:** Pipe directly into your `.env` during initial setup (review first!):
> ```bash
> python scripts/generate_secrets.py >> .env
> ```

## Per-Service Setup

### PostgreSQL

| Variable | Description |
|---|---|
| `DATABASE_URL` | Full DSN: `postgresql+asyncpg://user:password@host:5432/dbname` |
| `POSTGRES_USER` | Database user name (default: `swarm`) |
| `POSTGRES_PASSWORD` | Strong random password — generate with `generate_secrets.py` |
| `POSTGRES_DB` | Database name (default: `swarm`) |
| `POSTGRES_HOST` | Hostname — `postgres` inside Docker Compose, `localhost` otherwise |
| `POSTGRES_PORT` | Port (default: `5432`) |

**Steps:**

1. Generate a password: `python scripts/generate_secrets.py` → copy `POSTGRES_PASSWORD`.
2. Set `DATABASE_URL` using the same credentials:
   ```
   DATABASE_URL=postgresql+asyncpg://swarm:<POSTGRES_PASSWORD>@postgres:5432/swarm
   ```
3. For local dev the Docker Compose service handles database creation automatically.
4. For production use a managed service (RDS, Cloud SQL, Supabase) and load credentials
   from your secrets manager.

---

### Redis

| Variable | Description |
|---|---|
| `REDIS_URL` | Connection URL: `redis://[:password@]host:6379/db_number` |

**Steps:**

1. Local dev: default `redis://redis:6379/0` works with the bundled Compose service.
2. Production: use a managed Redis (ElastiCache, Upstash, Redis Cloud).  Set a strong
   `requirepass` and update `REDIS_URL` to include the password:
   ```
   REDIS_URL=redis://:your-strong-password@your-redis-host:6379/0
   ```

### JWT & Session Keys

| Variable | Description |
|---|---|
| `JWT_SECRET_KEY` | 64-char hex — HMAC key for signing JWT access and refresh tokens |
| `SECRET_KEY` | 64-char hex — signing key for sessions, cookies, and CSRF tokens |
| `ENCRYPTION_KEY` | 32-byte URL-safe — symmetric AES/Fernet key for stored PII |

These are all generated by `generate_secrets.py`.  Rules:

- Minimum length: 64 hex characters for JWT/session keys, 32 bytes for encryption key.
- Must be unique per environment — **never reuse a dev key in production**.
- Rotation schedule: JWT/session every 90 days, encryption key every 365 days (requires
  re-encrypting stored data — see [Rotating Secrets](#rotating-secrets)).

### Stripe (Payments)

| Variable | Description |
|---|---|
| `STRIPE_API_KEY` | Secret key — prefix `sk_test_` (test) or `sk_live_` (prod) |
| `STRIPE_PUBLISHABLE_KEY` | Publishable key — prefix `pk_test_` or `pk_live_` |
| `STRIPE_WEBHOOK_SECRET` | Webhook signing secret — prefix `whsec_` |
| `STRIPE_TEST_MODE` | `TRUE` to use test keys, `FALSE` for live keys |

**Where to get keys:**

1. Log in at [https://dashboard.stripe.com](https://dashboard.stripe.com).
2. Navigate to **Developers → API keys**.
3. Copy the **Secret key** (`sk_…`) → `STRIPE_API_KEY`.
4. Copy the **Publishable key** (`pk_…`) → `STRIPE_PUBLISHABLE_KEY`.
5. Navigate to **Developers → Webhooks → Add endpoint**.
6. Enter your backend webhook URL: `https://api.realms2riches.com/webhooks/stripe`.
7. Select events: `checkout.session.completed`, `customer.subscription.*`, `invoice.*`.
8. After saving, click **Reveal signing secret** → `STRIPE_WEBHOOK_SECRET`.

> **Test vs Live:** Keep `STRIPE_TEST_MODE=TRUE` and use `sk_test_` / `pk_test_` keys until
> you are ready for live transactions.  Switch all three keys together atomically.

### SMTP (Email)

| Variable | Description |
|---|---|
| `SMTP_USER` | Login email address (e.g. `ops@realms2riches.com`) |
| `SMTP_PASS` | Password or app-specific password |
| `SMTP_SERVER` | SMTP hostname (e.g. `smtp.gmail.com`, `smtp.sendgrid.net`) |
| `SMTP_PORT` | `587` for STARTTLS (recommended) · `465` for SSL · `25` for plain |
| `CONTACT_EMAIL` | Public contact address shown in outbound emails |

**Provider-specific instructions:**

#### Gmail / Google Workspace
1. Enable 2-Step Verification on the Google account.
2. Go to **Account → Security → App passwords**.
3. Create an app password for "Mail" / "Other (SwarmOS)".
4. Use that 16-char password as `SMTP_PASS`.
5. Set `SMTP_SERVER=smtp.gmail.com` and `SMTP_PORT=587`.

#### SendGrid
1. Log in at [https://app.sendgrid.com](https://app.sendgrid.com).
2. Navigate to **Settings → API Keys → Create API Key**.
3. Choose **Restricted Access** with **Mail Send** permission.
4. Set `SMTP_USER=apikey` (literal string) and `SMTP_PASS=<your-sendgrid-api-key>`.
5. Set `SMTP_SERVER=smtp.sendgrid.net` and `SMTP_PORT=587`.

#### AWS SES
1. Verify your sending domain in the SES console.
2. Create SMTP credentials under **SES → Account dashboard → SMTP settings**.
3. Use the generated IAM credentials as `SMTP_USER` and `SMTP_PASS`.
4. Set `SMTP_SERVER=email-smtp.<region>.amazonaws.com` and `SMTP_PORT=587`.

### Encryption Key

| Variable | Description |
|---|---|
| `ENCRYPTION_KEY` | 32-byte URL-safe base64 — used for encrypting sensitive database fields |

Generated by `generate_secrets.py`.  This key is used with a symmetric cipher (Fernet/AES-256)
to encrypt PII stored in the database.  Losing this key means losing access to encrypted data —
**back it up in your secrets manager immediately after generation**.

## Validating Your Configuration

After filling in `.env`, run:

```bash
python scripts/validate_env.py
```

The script checks every critical variable and prints a clear pass/fail report:

```
✅  Database (PostgreSQL)
     ✅  DATABASE_URL         Full Postgres connection string
     ✅  POSTGRES_PASSWORD    Postgres user password
     ...

❌  JWT / Auth (Security)
     ❌  JWT_SECRET_KEY       64-char hex secret for JWT signing
     ✅  SECRET_KEY           64-char hex secret for session / cookie signing
```

Exit codes:

| Code | Meaning |
|---|---|
| `0` | All critical variables are set |
| `1` | One or more critical variables are missing or empty |

Use in CI/CD pre-deploy gate:

```yaml
# .github/workflows/deploy.yml
- name: Validate environment
  run: python scripts/validate_env.py --no-color
```

## Rotating Secrets

Follow this procedure whenever a secret must be rotated (scheduled or after a suspected
compromise).  This mirrors the SOP pattern used across SwarmEnterprise v2 operations.

### Standard rotation (JWT_SECRET_KEY, SECRET_KEY, POSTGRES_PASSWORD)

```
1. GENERATE  → python scripts/generate_secrets.py   (produces new value)
2. STAGE     → Add NEW_<KEY>=<new_value> to secrets manager alongside the old value
3. DEPLOY    → Roll out new application version that accepts both old and new
               (dual-validation window — typically 15 minutes for JWT)
4. CUTOVER   → Remove OLD value; set <KEY>=<new_value> as the sole active secret
5. VERIFY    → python scripts/validate_env.py
6. AUDIT     → Record rotation date, operator, and reason in your change log
```

### Encryption key rotation (ENCRYPTION_KEY)

Encryption key rotation requires re-encrypting stored data — coordinate with engineering:

```
1. Generate new key with generate_secrets.py
2. Run the re-encryption migration:
      python manage.py rotate_encryption_key \
          --old-key $OLD_ENCRYPTION_KEY \
          --new-key $NEW_ENCRYPTION_KEY
3. Verify all records decrypt successfully in staging before production
4. Remove old key from secrets manager after successful verification
```

### Emergency rotation (suspected compromise)

1. **Immediately** invalidate the old secret in your secrets manager.
2. Generate a new value: `python scripts/generate_secrets.py`.
3. Deploy the new value — this will invalidate all active sessions (users must re-login).
4. Notify the security contact: [robertdemottojr50@gmail.com](mailto:robertdemottojr50@gmail.com).
5. Review access logs for the compromised credential.
6. Document the incident in the incident register.

### Rotation schedule

| Secret | Frequency | Notes |
|---|---|---|
| `JWT_SECRET_KEY` | 90 days | Invalidates all active JWTs |
| `SECRET_KEY` | 90 days | Invalidates sessions/cookies |
| `POSTGRES_PASSWORD` | 180 days | Coordinate with DB admin |
| `STRIPE_API_KEY` | On compromise only | Rotate in Stripe dashboard |
| `SMTP_PASS` | 180 days or on compromise | |
| `ENCRYPTION_KEY` | 365 days | Requires data re-encryption migration |
| `CLOUDFLARE_TUNNEL_TOKEN` | 365 days | |

## Security Best Practices

### ❌ Never do this

```bash
git add .env                          # Never commit .env
git commit -m "add secrets"           # Never commit secrets in any file
echo $SECRET_KEY                      # Avoid printing secrets in logs
curl ... -d "key=$STRIPE_API_KEY"     # Never send secrets in URLs
```

### ✅ Always do this

```bash
# .gitignore must contain:
.env
*.env
.env.*
!.env.example

# Verify .env is not tracked:
git ls-files .env         # should return nothing
git status --short .env   # should show nothing
```

### Local development

- Use `.env` only for local development.
- Never use production secrets locally — use test/sandbox credentials.
- Use `STRIPE_TEST_MODE=TRUE` and Stripe test keys (`sk_test_`, `pk_test_`).
- Use a local Postgres and Redis — not production databases.

### CI/CD (GitHub Actions)

Store secrets in **GitHub Actions Secrets** (`Settings → Secrets and variables → Actions`):

```yaml
env:
  DATABASE_URL:           ${{ secrets.DATABASE_URL }}
  JWT_SECRET_KEY:         ${{ secrets.JWT_SECRET_KEY }}
  SECRET_KEY:             ${{ secrets.SECRET_KEY }}
  STRIPE_API_KEY:         ${{ secrets.STRIPE_API_KEY }}
  STRIPE_WEBHOOK_SECRET:  ${{ secrets.STRIPE_WEBHOOK_SECRET }}
```

### Production

- Use a dedicated secrets manager: **HashiCorp Vault**, **AWS Secrets Manager**, or
  **GCP Secret Manager**.
- Never bake secrets into Docker images.
- Inject secrets as environment variables at runtime via your orchestrator (ECS task
  definitions, Kubernetes secrets, Docker Swarm secrets).
- Enable audit logging on your secrets manager to track all reads and writes.
- Apply least-privilege: each service should only access the secrets it needs.
- Scan your codebase regularly with `git-secrets` or `trufflehog`:
  ```bash
  trufflehog git file://. --only-verified
  ```

### Secret hygiene checklist

- [ ] `.env` is in `.gitignore`
- [ ] No secrets in `docker-compose.yml` (use `env_file:` or secrets manager)
- [ ] `JWT_SECRET_KEY` and `SECRET_KEY` are ≥ 64 hex characters
- [ ] `POSTGRES_PASSWORD` is random, not a dictionary word
- [ ] Stripe keys match environment (`sk_test_` for dev, `sk_live_` for prod)
- [ ] Rotation schedule is documented and calendared
- [ ] Secrets are backed up in a secrets manager before use in production

## Environment Promotion

```
Local Dev          Staging                 Production
─────────          ───────                 ──────────
.env file          GitHub Actions Secrets  HashiCorp Vault / AWS SM
sk_test_ keys      sk_test_ keys           sk_live_ keys
local Postgres     managed Postgres        managed Postgres (HA)
STRIPE_TEST=TRUE   STRIPE_TEST=TRUE        STRIPE_TEST=FALSE
separate secrets   separate secrets        separate secrets
```

**Rule:** Secrets must never flow upward.  A local `.env` value must never be used in staging
or production.  Generate fresh secrets for each environment with `generate_secrets.py`.
